# Systematic Debugging

## 개요

무작위 수정은 시간을 낭비하고 새 문제를 만든다.

핵심 원칙: **근본 원인 확정 전에 수정 제안 금지. 증거 먼저, 주장은 그 다음.**

## 철칙

```
근본 원인 없이 수정하지 않는다.
```

해당하는 모든 상황:
- 코드·스크립트 오류
- 데이터 파이프라인 불일치
- memory/JSON 파싱 오류
- 세션 체크리스트 항목 미완료
- 예상과 다른 출력

**특히 이럴 때 반드시 사용:**
- 시간 압박이 있을 때 (급할수록 추정이 유혹적)
- "간단한 수정"처럼 보일 때
- 이미 여러 번 수정을 시도했을 때

## 4단계 프로세스

각 단계를 완료하기 전에 다음 단계로 넘어가지 않는다.

### 1단계: 근본 원인 조사 (Root Cause)

**수정 시도 전 반드시:**

1. **실제 출력 읽기** — 에러 메시지·로그를 건너뛰지 않는다. 스택 트레이스 전체 읽기.
2. **재현 확인** — 이 문제를 안정적으로 재현할 수 있는가? 매번 발생하는가?
3. **최근 변경 확인** — 이 문제를 유발할 수 있는 최근 변경은 무엇인가?
4. **다중 컴포넌트 진단** — 시스템이 여러 계층(스크립트 → 파일 → DB → 출력)으로 구성된 경우:
   각 경계에서 입력·출력을 기록하여 어느 계층에서 깨지는지 확인한다.

### 2단계: 패턴 분석 (Pattern)

1. **작동하는 유사 사례 찾기** — 같은 코드베이스에서 비슷하게 작동하는 것은?
2. **레퍼런스 완독** — 패턴을 구현 중이라면 레퍼런스를 끝까지 읽는다. 훑어보기 금지.
3. **차이점 목록화** — 작동하는 것과 작동하지 않는 것의 차이를 모두 나열한다.

### 3단계: 가설과 검증 (Hypothesis)

1. **단일 가설 수립** — "X가 근본 원인이다. 왜냐하면 Y이기 때문." 명확히 기술.
2. **최소 변경으로 테스트** — 가설을 검증할 가장 작은 변경만.
3. **한 번에 하나** — 복수 변경 동시 시도 절대 금지.
4. **모른다면 말한다** — "X를 이해하지 못한다"고 명시. 추정으로 진행하지 않는다.

### 4단계: 수정 구현 (Fix)

1. **증거 확보** — 수정 전 실패 상태, 수정 후 성공 상태를 각각 기록.
2. **단일 수정** — 근본 원인에 대한 하나의 변경만.
3. **검증** — 실제 실행 후 출력 확인. "작동할 것 같다"는 완료가 아니다.
4. **3회 실패 시 중단** — 3번 이상 수정해도 해결되지 않으면:
   → 구현 문제가 아닌 **구조 문제**. Arki에게 에스컬레이션.

## 레전드팀 적용 사례

| 증상 | 조사 시작점 |
|---|---|
| JSON 파싱 오류 | 파일 직접 열어서 구문 확인, 마지막 수정 시점 확인 |
| 세션 체크리스트 항목 미완료 | 해당 파일 실제 존재 여부 확인 (ls/Read) |
| 스크립트 실행 실패 | 에러 메시지 전문 읽기, node_modules 존재 여부 확인 |
| memory 업데이트 미반영 | 파일 직접 읽어서 내용 확인 (작성 완료 ≠ 저장 완료) |
| git push 미반영 | auto-push.js 출력 확인, remote 상태 확인 |
| 예상과 다른 출력 | 실제 출력 기록 후 기대 출력과 비교 |

## 흔한 합리화

| 변명 | 현실 |
|---|---|
| "간단한 문제라서 프로세스 불필요" | 간단한 문제도 근본 원인이 있다. 프로세스는 간단한 버그엔 빠르다. |
| "급해서" | 체계적 디버깅이 추정·수정·재시도보다 빠르다. |
| "일단 바꿔보고" | 첫 번째 수정이 패턴을 결정한다. 처음부터 올바르게. |
| "이전에 같은 방식으로 했으니까" | 이번 케이스를 검증하라. |
| "한 번만 더 시도" (2번 이미 실패) | 3번 실패 = 구조 문제. Arki 에스컬레이션. |

## 완료 선언 형식

```
✅ 디버깅 완료
근본 원인: [원인 한 줄]
수정 내용: [변경 한 줄]
검증: [실행 명령] → [실제 출력]
```
