# Publish Contract — Legend Team v0.3.0

This document defines the contract between the write channel (Claude Code / local file system) and the read channel (Cloudflare Pages viewer). The viewer must only depend on fields defined here.

---

## 2-Plane Architecture

| Plane | Path | Owner | Access |
|-------|------|-------|--------|
| Control plane | `topics/{topicId}/` | Claude Code (write) | Local only |
| Artifact plane | `reports/{YYYY-MM-DD}_{slug}/` | Claude Code (write) | Viewer reads |
| Published manifest | `dist/data/published/` | `build.js` (generated) | Viewer reads |

The viewer **must not** depend on `controlPath` or any `topics/` paths. All viewer data flows through `published/`.

---

## topics_manifest.json

**Path:** `dist/data/published/topics_manifest.json`
**Generated by:** `node scripts/build.js`
**Source:** `memory/shared/topic_index.json`

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `generatedAt` | ISO timestamp | yes | Build time |
| `topics` | array | yes | List of topic entries |
| `topics[].id` | string | yes | Canonical topic ID (e.g. `topic_003`) |
| `topics[].title` | string | yes | Human-readable title |
| `topics[].status` | string | yes | One of: `open`, `in-progress`, `review`, `suspended`, `closed` |
| `topics[].created` | string (YYYY-MM-DD) | yes | Topic creation date |
| `topics[].reportPath` | string \| null | yes | Path to artifact directory (e.g. `reports/2026-04-04_local-vs-server`) |
| `topics[].reportFiles` | string[] | yes | Filenames within `reportPath` (e.g. `["ace_rev01.md"]`) |
| `topics[].published` | boolean | yes | Whether this topic is visible in the viewer |
| `topics[].outcome` | string \| null | optional | Summary of closed topic outcome |
| `topics[].note` | string \| null | optional | Freeform annotation |

### Fields NOT included (control-plane only)

- `controlPath` — local workspace path, not exposed to viewer
- `masterDecisions` — decision IDs, redundant with decisions_summary.json

---

## decisions_summary.json

**Path:** `dist/data/published/decisions_summary.json`
**Generated by:** `node scripts/build.js`
**Source:** `memory/shared/decision_ledger.json`

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `generatedAt` | ISO timestamp | yes | Build time |
| `decisions` | array | yes | All entries from decision_ledger.json |

The `decisions` array is passed through as-is. Fields within each decision entry follow the `decision_ledger.json` schema.

---

## Report Files

**Path pattern:** `reports/{YYYY-MM-DD}_{slug}/{role}_rev{n}.md`
**Fetched by:** `DataLoader.getReport(folderSlug, fileName)`

Report files use canonical v0.3.0 frontmatter (new topics) or legacy frontmatter (pre-v0.3.0 topics). The viewer renders them as markdown and does not parse frontmatter fields.

---

## Viewer Dependencies (canonical fields)

The viewer (`app/topic.html`, `app/index.html`, etc.) must only reference these fields:

- `topic.id` — for routing (`?id=topic_003`)
- `topic.title`, `topic.status`, `topic.created`
- `topic.reportPath` — derive `folderBase` for report fetching
- `topic.reportFiles` — enumerate report file names
- `topic.published` — filter visibility
- `topic.outcome`, `topic.note` — display metadata

**The viewer must NOT reference:**
- `topic.path` — legacy field, deprecated
- `topic.controlPath` — control-plane only
- `topic.masterDecisions` — use `decisions_summary.json` instead

---

## Canonical Frontmatter Schema (v0.3.0)

New role output files must use this frontmatter:

```yaml
---
topic: topic_005
topic_slug: my-topic-slug
role: ace
phase: framing
revision: 1
date: 2026-04-05
report_status: draft
session_status: open
accessed_assets:
  - topic_index.json
  - decision_ledger.json
---
```

### Canonical field values

| Field | Valid values |
|-------|-------------|
| `role` | `ace`, `arki`, `fin`, `riki`, `editor`, `nova`, `master` |
| `report_status` | `draft`, `reviewed`, `approved`, `superseded`, `speculative` |
| `session_status` | `open`, `in-progress`, `review`, `suspended`, `closed` |

### Legacy compatibility

Files created before v0.3.0 use `agent` (not `role`) and a single `status` field. `validate-output.ts` detects these automatically and issues warnings (not failures) unless `--strict` is passed.

## controlPath Policy

| Topic | controlPath | Meaning |
|-------|-------------|---------|
| topic_001 | `topics/topic_001` | Control plane exists |
| topic_002 | `null` | Retroactive — no local control plane (report-only topic) |
| topic_003 | `null` | Retroactive — no local control plane (report-only topic) |
| topic_004 | `null` | Session ended before control plane was created |
| topic_005+ | `topics/topic_005` | New topics always have control plane |

`controlPath: null` is intentional for retroactive/historical topics where only the report artifact plane exists. Scripts fall back to `topics/{topicId}` if `controlPath` is null or missing.

## npm Scripts

```
npm run build         → node scripts/build.js  (generates dist/ + published/)
npm run build:viewer  → alias for build
npm run validate      → ts-node scripts/validate-output.ts <file>
npm run report        → ts-node scripts/build-report.ts <topicId>
npm run debate        → ts-node scripts/run-debate.ts <topicId> <role> ...
npm run create-topic  → ts-node scripts/create-topic.ts "<title>" [slug]
```
