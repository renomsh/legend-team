# Ace — Scope 확인 질문 (S Grade, 오픈 탐색형)

## 전제 인식

S Grade이므로 ace-framing 스킬 미발동. Scope 확인 질문만 수행합니다.
Master의 방향 원문에서 읽히는 것: 시스템 이름 변경 + 3개 구조 변화(Nenux 신설, Ace 재정의, Zero 신설) + grade 체계 재정의.
이 토픽은 10세션+ 장기 탐색입니다. 첫 세션 Draft 수립이 목적.

## Q1. Nenux의 성격 — 코드 레벨인가, 메타-역할인가

**가설**: Nenux = 코드 레벨 Node.js 허브. 현재 Main(sonnet)이 담당하는 역할 디스패치·세션 관리·컨텍스트 브로킹을 Nenux 계층으로 분리하겠다는 것.

확인이 필요한 지점:
- "agent X"가 "특정 에이전트 페르소나가 없다"는 의미인지, "에이전트가 아닌 시스템 레이어"라는 의미인지
- 허브의 구체 형태 — **라우터** (grade 판별 → 역할 디스패치) / **컨텍스트 브로커** (역할 간 메모리 전달) / **세션 관리자** (open/close/turn 기록) 중 어느 것이 핵심인가. 셋 다인가
- 현재 pre-tool-use hook + Main이 이 역할을 분산 수행 중 — Nenux는 이를 단일 코드 모듈로 통합하는 것인가

> **Master 확인 요청**: Nenux = "페르소나 없는 시스템 레이어(코드)"가 맞는가? 아니면 역할 명세도 갖는가?

## Q2. Ace 뒤단 이동 — 프레이밍 역할은 어디로 가는가

**가설**: "앞단에서 뒤단"은 Ace가 topic open 직후 프레이밍 첫 발언을 하는 현재 구조를 폐기한다는 것. 대신 Ace는 모든 역할 발언을 들은 뒤 최종 조정자로만 등장.

확인이 필요한 지점:
- 프레이밍(scope 정의, 결정축, 가정 설정)은 누가 맡는가 — Nenux가 자동 처리? 첫 주자 역할(Arki? Nova?)이 흡수?
- "마스터 페르소나"의 의미 — Ace가 Master의 판단 대리인이라는 현재 정의 강화인가, 아니면 Ace = Master 목소리 그 자체로 격상(Ace 발언 = Master 발언으로 처리)되는가
- 현재 Ace는 Grade A/B에서 framing + 종합검토 두 번 발언. 뒤단 이동 후 framing은 제거, 종합검토+결정만 남기는 구조인가

> **Master 확인 요청**: 프레이밍 역할의 귀속처를 확인해 주십시오. Ace가 완전히 빠지는가, 아니면 S grade처럼 scope 확인만 하는 경량 모드로 전환되는가?

## Q3. Zero의 활성화 조건 — 모든 구현에 기본인가, 키워드 트리거인가

**가설**: Zero = 현재 D grade(bug/fix/patch 키워드) + 구현 완료 후 코드 품질 리뷰 단계에서 호출. 별도 grade 신설보다 기존 C/D grade에 Zero를 삽입하는 방식.

확인이 필요한 지점:
- Zero의 호출 시점 — 구현 전(설계 단계에서 기존 하드코딩 탐지) / 구현 중(Dev와 페어) / 구현 후(리팩토링 감사)
- 호출 grade 범위 — D만인가, C/D 모두인가, 모든 grade의 구현 단계에 기본 포함인가
- Zero와 Dev의 관계 — Dev가 구현하고 Zero가 감사하는 직렬 구조인가, 동시 호출 병렬인가
- "클린코드 리팩토링"이 Zero의 독립 출력물(별도 PR/파일)인지 Dev 출력물에 인라인 수정인지

> **Master 확인 요청**: Zero를 모든 구현 세션에 기본 포함할 것인가, 아니면 특정 조건(키워드/grade)에서만 호출할 것인가?

## Q4. Grade 재정의 — 재정의의 핵심 변화는 무엇인가

**가설**: Nenux+Ace 구조 변화로 인해 "첫 주자"가 grade마다 달라짐. 현재 A/B/C/D 모두 Ace(또는 Dev) 첫 주자 → Nenux 도입 후 Nenux가 grade 판별 후 첫 역할을 다르게 지정.

확인이 필요한 지점:
- "호출순서·방식" 재정의가 핵심 변화인지, 아니면 grade 정의(조건·규모·성격) 자체도 바뀌는지
- Zero가 추가됨에 따라 D grade 정의 변화 — 현재 "Dev 직행" → "Dev → Zero" 구조로 바뀌는가
- S grade 재정의 필요 여부 — 현재 S = "Ace-Nova-Arki-Riki 꼬리물기 루프". Ace 뒤단 이동 후 S grade 첫 주자는 누구인가
- 새로운 grade 신설 여부 (예: Zero 전용 grade E)

> **Master 확인 요청**: grade 재정의가 "역할 호출 순서 조정"에 그치는가, 아니면 grade 조건/수 자체도 변경하는가?

## Q5. 이번 세션 Draft의 형태

**가설**: CLAUDE.md 수정 초안 + 역할 명세 문서(각 역할별 1페이지) + 결정 박제(D-xxx) 3~5건. 코드/구현은 다음 세션.

확인이 필요한 지점:
- Draft = 개념 문서(방향 합의)인가, 구현 가능한 명세(즉시 코딩 착수 가능)인가
- 이번 세션에서 박제(D-xxx)할 결정의 수와 수준 — 큰 방향 3개인가, 세부 정책 10개인가
- 참조 링크(https://ecc.tools/, https://goddaehee.tistory.com/575)에서 구체적으로 가져올 개념이 있는가 — Ace가 사전에 확인하고 컨텍스트 제공이 필요한가
- Edi 최종 산출물 형태 — CLAUDE.md 개정안? 별도 `NENUX.md`? `agents/` 디렉토리 역할 파일?

> **Master 확인 요청**: 이번 세션 산출물을 한 줄로 정의해 주십시오. "~을 결정하고, ~형태로 문서화한다."
